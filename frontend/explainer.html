How we converted the NDIS Pricing Arrangements and Price Limits and Support Catalogue from Word and Excel files into structured, queryable, API-accessible information.

The starting point

The NDIS pricing artefacts — the Pricing Arrangements and Price Limits (PAPL) and Support Catalogue — are published as a Word document and an Excel spreadsheet. They are among the most downloaded documents on the NDIS website.

While these formats are familiar, they create a number of limitations:

  • Users must download and manually search large files to find the information they need
  • Every update requires re-publishing complete new versions of each file
  • Version history is managed through filename conventions (e.g. v1.1_0.docx) rather than a proper audit trail
  • Provider portals and other systems that need pricing data must re-key it manually — there is no machine-readable source
  • There is no way to filter, compare, or query the data without opening the file

The digital-first approach

In a digital-first model, structured data is the master copy. The Word document and Excel spreadsheet become outputs — generated from the source data — rather than the source of truth themselves.

This exemplar uses three structured formats, each suited to a different type of content:

JSON

Support Catalogue

Each of the 635 current support items is represented as a structured JSON object with typed fields for price limits by jurisdiction, claiming rules, registration group, unit, and effective dates.

This makes the data immediately queryable — filter by category, search by name, look up a single item by number.

Markdown

Pricing Arrangements text

The narrative and policy content of the PAPL — introductions, explanations, claiming rules — is stored as Markdown. Each top-level section becomes a separate file.

Markdown is human-readable as plain text, renderable as HTML for the web, and convertible back to Word or PDF when needed.

YAML

Document structure

The outline and metadata of each document — section titles, slugs, headings, and (for the Code Guide) which support categories each section applies to — is stored as YAML.

This powers the sidebar navigation and the contextual Code Guide integration without any hardcoding in the application.

How the conversion works

This exemplar was built by extracting content from the existing source files using Python scripts. This is a one-time process each time a new version is published.

1

Support Catalogue extraction

The Excel file is read row by row. Each support item becomes a structured JSON object. Price limits are normalised into a per-jurisdiction map. Claiming rule flags (Y/N/NA) are converted to typed booleans. Reference tables for categories and registration groups are deduplicated automatically.

Script: scripts/extract_catalogue.py → data/support_catalogue.json

2

PAPL document extraction

The Word document is parsed paragraph by paragraph. Heading styles (Heading 1–5) become Markdown headings. Tables are converted to Markdown table syntax. Each top-level section (Heading 1) becomes its own Markdown file, and the document outline is written to a YAML index.

Script: scripts/extract_papl.py → data/papl/*.md + data/papl_structure.yaml

3

AT & HM Code Guide extraction

The same process is applied to the Assistive Technology, Home Modifications and Consumables Code Guide. An additional metadata field — relevant_categories — maps each section to the support categories it covers (03, 05, 06). This powers the contextual Code Guide panel.

Script: scripts/extract_atcg.py → data/atcg/*.md + data/atcg_structure.yaml

4

API layer

A FastAPI application loads all structured data into memory at startup and exposes it through a REST API with full OpenAPI documentation. Data is filterable, searchable, and accessible per-item — capabilities that are impossible with a static file download.

See API documentation →

5

Web interface

The same API powers this website. The Support Catalogue browser, PAPL reader, and item detail pages all fetch data from the API at runtime — the same endpoints available to any external system or provider wanting programmatic access.

What this makes possible

🔁

Single source of truth

The structured data files are the authoritative version. The website, API, and any future static document exports are all generated from the same source — eliminating dual-maintenance.

📋

Proper version control

Changes to pricing data are tracked in Git. Every update is a reviewable diff. The full history of what changed, when, and why is available — far more rigorous than filename-based versioning.

🔌

System integration

Provider portals, participant tools, and internal systems can query pricing data directly via the API — no re-keying, no risk of transcription errors, always current.

🗂️

Contextual surfacing

The Code Guide toggle demonstrates how multiple documents can be surfaced contextually — the right guidance at the right moment — rather than requiring users to locate and cross-reference separate PDFs.

What this exemplar is not

This is a proof of concept, not a production system. It demonstrates the approach and the user experience, but a production implementation would require:

  • Formal content governance — who can update the source data, what review process applies
  • Security review of the API (authentication, rate limiting, access controls)
  • Integration with existing publishing and CMS workflows
  • Accessibility audit of the web interface
  • Legal review of content representation and liability for data accuracy

The exemplar is intended to reduce uncertainty about whether the approach is feasible — not to be the final implementation.
